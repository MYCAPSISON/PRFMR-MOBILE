--------------------------------------------------------------------------------
  SESSION 2 — Authentication Bug Investigation & Fixes
--------------------------------------------------------------------------------

PROBLEM: After login, all API requests returned 401 Not Authenticated.
Users could not see any data.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  BUG #1: Fetch API hides Set-Cookie headers
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ROOT CAUSE:
  React Native's fetch() copies the browser security model: it deliberately
  hides Set-Cookie response headers from JavaScript. After login, calling
  response.headers.get("set-cookie") returned null, even though the server
  was sending session cookies. Our code thought no cookies were received,
  so nothing was saved, so all subsequent requests went out with no cookies.

FIX APPLIED:
  Replaced the fetch() login call with XMLHttpRequest (XHR).
  XHR's getAllResponseHeaders() returns ALL headers including Set-Cookie.
  The login path changed to loginWithXhr() in lib/api.ts.

FILE CHANGED: lib/api.ts
  - Added loginWithXhr() function using XMLHttpRequest

FILE CHANGED: context/AuthContext.tsx
  - Updated login() to call loginWithXhr() instead of apiFetch("/auth/login")

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  BUG #2: GAESA cookie overwrote connect.sid
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ROOT CAUSE:
  After login, when /user/me was called, its response set a new cookie GAESA
  (a secondary token). Our code saved GAESA as the new session, discarding
  connect.sid. All future requests sent only GAESA → 401.

FIX APPLIED:
  Implemented a cookie jar (dictionary) that accumulates all cookies by name.
  Adding a new cookie MERGES it in rather than replacing everything.
  Jar key changed from "prfmr_session" to "prfmr_jar" in SecureStore.

FILE CHANGED: lib/api.ts
  - cookieJar changed from string to Record<string, string>
  - addCookie(), buildCookieHeader(), saveJar(), loadJar() updated

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  BUG #3: Manual Cookie header blocked iOS native cookie jar
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ROOT CAUSE:
  connect.sid is HttpOnly — invisible to ALL JavaScript. iOS stores it in
  NSHTTPCookieStorage. When we manually set Cookie: GAESA in fetch(), iOS
  used ONLY that manual header, ignoring NSHTTPCookieStorage entirely, so
  connect.sid was never sent → 401 on all requests.

FIX APPLIED:
  Removed manual Cookie header from apiFetch() entirely.
  Only use credentials: "include", which tells iOS to send ALL cookies
  from NSHTTPCookieStorage automatically, including connect.sid.

FILE CHANGED: lib/api.ts
  - Removed: reqHeaders["Cookie"] = buildCookieHeader()
  - Kept: credentials: "include" in all fetch() calls
  - Updated log format to show "native jar: handles connect.sid"

  SESSION 3 — Food Logging Bug Fixes + Nutrition Modal Rebuild
--------------------------------------------------------------------------------

PROBLEMS REPORTED:
  1. Today's food logs not showing in Dashboard (meals section always empty)
  2. POST /food returning 400 Validation failed
  3. Nutrition modal only had Search — needed Search, Barcode, Custom, Whole Food

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  BUG #4: Food logs not showing — wrong field name
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ROOT CAUSE:
  The server returns food entries with the field name "meal".
  The mobile app's FoodEntry interface used "mealType".
  The grouping code filtered on e.mealType === mt — always empty since the
  actual field was e.meal. Every meal section showed "Nothing logged."

FIX: app/(tabs)/index.tsx
  - FoodEntry.mealType → FoodEntry.meal
  - entries.filter(e => e.mealType === mt) → entries.filter(e => e.meal === mt)

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  BUG #5: POST /food 400 Validation failed — wrong payload shape
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

ROOT CAUSE:
  Mobile app was sending payloads with wrong/missing fields:
  - Used key "mealType" — server expects "meal"
  - Missing "grams" (required positive number)
  - Missing "macroSource" (required: "manual")
  - Missing "microSource" (required: "none")
  - Missing "fibre" (defaults to 0 if absent but causes issues)
  - sourceType was "database" — server expected "off" for Open Food Facts items

  WRONG payload (what we were sending):
    { name, calories, protein, carbs, fat, date, mealType: "lunch", sourceType: "database" }

  CORRECT payload (what the server needs):
    { name, calories, protein, carbs, fat, fibre, grams, date, meal: "lunch",
      sourceType: "off", macroSource: "manual", microSource: "none" }

FIX: app/(tabs)/index.tsx (Dashboard modal)
  - All POST payloads: mealType → meal
  - Added: grams: 100 (default 100g serving)
  - Added: macroSource: "manual"
  - Added: microSource: "none"
  - Added: fibre: 0 (manual) or item.fibre (search)
  - Changed sourceType: "database" → "off" for Open Food Facts items

FIX: app/(tabs)/nutrition.tsx (Nutrition tab modal)
  - Same payload fixes applied
  - Added buildPayload() helper that always constructs the correct full payload

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
