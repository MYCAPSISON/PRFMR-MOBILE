# PRFMR — React Native / Expo UI Specification
**For: Mobile agent rebuilding the web UI in React Native + Expo**

This document covers everything you need to recreate the PRFMR web UI as a native mobile app — design tokens, navigation architecture, screen-by-screen layout, component patterns, API integration, and platform-specific notes.

---

## 1. DESIGN SYSTEM

### 1.1 Colour Tokens

The web app uses HSL custom properties. Below are the resolved hex/rgba equivalents for React Native `StyleSheet`:

| Token | Role | Hex |
|---|---|---|
| `background` | Screen background | `#0f1117` (hsl 220 20% 7%) |
| `foreground` | Primary text | `#eceef2` (hsl 210 10% 95%) |
| `card` | Card / panel background | `#13161d` (hsl 220 16% 11%) |
| `secondary` | Subtle surface / input bg | `#181c26` (hsl 220 15% 13%) |
| `muted` | Dividers, skeleton bg | `#1e222e` (hsl 220 12% 17%) |
| `muted-foreground` | Placeholder, secondary text | `#6b7280` (hsl 215 10% 50%) |
| `border` | Card / input borders | `#1a1e28` (hsl 220 10% 15%) |
| `primary` | Accent — orange | `#ff7a00` (hsl 24 100% 50%) |
| `primary-foreground` | Text on primary | `#ffffff` |
| `destructive` | Error / danger | `#c0392b` (hsl 0 62% 50%) |
| `sidebar-background` | Sidebar / bottom nav bg | `#0d1015` (hsl 220 18% 8%) |

**Status colours (consistent across all badge/indicator usage):**
```
green-400   = #4ade80   (good / on track / high)
yellow-400  = #facc15   (moderate / warning)
orange-400  = #fb923c   (aggressive / elevated)
red-400     = #f87171   (danger / poor / very high)
blue-300    = #93c5fd   (provisional / informational)
emerald-500 = #10b981   (elite AMQS tier)
amber-500   = #f59e0b   (good AMQS tier)
```

### 1.2 Typography

The web app uses three font families:
- **Body / UI text:** `Inter` — all labels, values, descriptions, form inputs
- **Display / headers:** `Space Grotesk` (web uses `Playfair Display` for h1–h6, but the app-wide modal/card headers use `Space Grotesk` via `font-display` class) — use for screen titles, card section headers, modal titles
- **Monospace:** `JetBrains Mono` — version stamps, any numeric debugging display

In Expo, load these via `expo-font` or `@expo-google-fonts/inter` etc.

**Font sizes and weights in use:**
```
Screen title:       20–24sp, Space Grotesk, weight 600
Card section h2:    14–16sp, Space Grotesk, weight 600
Body text:          14sp, Inter, weight 400
Secondary / meta:   12sp, Inter, weight 400, color muted-foreground
Badge text:         10–11sp, Inter, weight 500, uppercase
Value / number:     18–28sp, Inter or Space Grotesk, weight 600–700
Tiny label:         9–10sp, Inter, weight 500
```

### 1.3 Spacing & Radius

- **Screen horizontal padding:** 16px
- **Card internal padding:** 16px
- **Gap between cards:** 12–16px
- **Border radius — cards:** 9px (0.5625rem)
- **Border radius — buttons:** 6px (0.375rem)
- **Border radius — small chips/badges:** 4px
- **Minimum tap target height:** 48px (apply to all Pressable/TouchableOpacity)

### 1.4 Elevation / Shadows

Cards use a layered shadow for depth. In RN:
```js
cardShadow: {
  shadowColor: '#000',
  shadowOffset: { width: 0, height: 2 },
  shadowOpacity: 0.3,
  shadowRadius: 6,
  elevation: 4, // Android
}
```
Pressed state: scale 0.985, reduced shadow opacity.

### 1.5 Progress Bars

Custom height (not default): `height: 6` or `height: 8` depending on context. Animated width transition (0.8s cubic). Colour of fill bar maps to status (green/yellow/orange/red).

## 3. AUTHENTICATION FLOW

**Backend:** Session-based (cookie). All API calls must include credentials.

In React Native, manage sessions using `react-native-cookies` or a fetch wrapper that stores and sends the `connect.sid` cookie via `AsyncStorage` / `expo-secure-store`.

**Login screen layout:**
- Full-screen dark background
- Top: 48×48 orange rounded square with `Activity` icon (primary colour fill)
- Title: "Welcome back" — 28sp, Space Grotesk, weight 700
- Subtitle: muted-foreground, 14sp
- "Continue with Google" — full-width outlined button, 52px tall, Google icon left
- Divider: horizontal line with "or" centred
- "Email or Username" text input — 52px tall, bordered
- "Password" input — 52px tall, show/hide toggle button (eye icon) on right
- "Sign In" primary button — full width, 52px, orange background
- "Forgot password?" link below
- "Don't have an account? Request an invite" link at bottom

On mobile, Google OAuth opens an in-app browser (use `expo-auth-session` or `expo-web-browser` for the OAuth redirect, or simply call the backend OAuth route which redirects).

**Signup screen:**
- Invite code field (required — backend validates)
- Email, username, password fields
- Password requirements displayed inline (8+ chars, uppercase, number, special)

**Onboarding (4-step wizard):**

Step indicators: "Basics", "Body Metrics", "Lifestyle", "Goal" — displayed as horizontal step dots or numbered pills at the top.

- **Step 0 (Basics):** Age (number), Gender (radio: Male/Female/Other), Height in cm (number)
- **Step 1 (Body Metrics):** Current weight in kg (number), body fat % (optional)
- **Step 2 (Lifestyle):** Activity level (Select: Sedentary / Lightly Active / Moderately Active / Very Active / Extremely Active), Experience level (Select: Beginner / Intermediate / Advanced)
- **Step 3 (Goal):** Goal type radio cards — Fat Loss (flame icon), Maintenance (scale icon), Muscle Gain (zap icon) — each as a bordered card that fills with a subtle orange tint when selected

Navigation: Back/Next buttons at bottom. Final step submits to `POST /api/onboard`.

## 6. API INTEGRATION

### 6.1 Base Configuration

```js
const API_BASE = process.env.EXPO_PUBLIC_API_URL; // e.g. https://app.prfmr.link
```

All requests must include cookies. Use a wrapper:

```js
async function apiRequest(method, path, body) {
  const res = await fetch(`${API_BASE}${path}`, {
    method,
    credentials: 'include',
    headers: body ? { 'Content-Type': 'application/json' } : {},
    body: body ? JSON.stringify(body) : undefined,
  });
  if (!res.ok) throw new Error(await res.text());
  return res.json();
}
```

For React Native, `credentials: 'include'` alone does not persist cookies. Use `react-native-cookies` (`@react-native-cookies/cookies`) to manually store and re-attach the session cookie from the `Set-Cookie` response header.

### 6.2 Key API Endpoints

**Auth:**
- `POST /api/login` — body: `{ identifier, password }` — returns user object
- `POST /api/logout` — clears session
- `GET /api/user` — current authenticated user (or 401)
- `GET /api/me/profile` — full profile with calculated targets

**Dashboard / Food:**
- `GET /api/me/food-log/:date` — food entries for a date

### 6.3 State Management

Use **TanStack Query** (`@tanstack/react-query`) — it works in React Native. The same query key conventions as the web app apply:

```js
// Query key pattern: ['/api/path', date] for date-specific queries
useQuery({ queryKey: ['/api/me/food-log', date], queryFn: ... })

// Always invalidate related queries after mutations
queryClient.invalidateQueries({ queryKey: ['/api/me/food-log', date] })
```

### 6.4 Offline / Error Handling

- Show explicit error states (not silent fallbacks)
- Network errors: toast notification
- 401 responses: redirect to login, clear stored session cookie
- Loading states: skeleton loaders (not spinners alone)

## 7. PLATFORM-SPECIFIC NOTES

### 7.1 Safe Areas

Wrap root in `<SafeAreaProvider>` from `react-native-safe-area-context`. Add `useSafeAreaInsets()` to bottom tab bar and any bottom sheets. The web app already handles `env(safe-area-inset-bottom)` — replicate with `insets.bottom`.

### 7.2 Keyboard

Use `KeyboardAvoidingView` on all screens with text inputs. On iOS use `behavior="padding"`, on Android use `behavior="height"`.

### 7.3 Animations

The web app uses Framer Motion heavily. In React Native:
- Use **Reanimated 2/3** (`react-native-reanimated`) for enter/exit animations and spring transitions
- Use `useSharedValue`, `withSpring`, `withTiming` for animated values
- Card press → `withSpring(0.985)` scale, release → `withSpring(1.0)`
- Progress bar fill → `withTiming(targetWidth, { duration: 800 })`
- Modals / sheets → slide up from bottom with spring

### 7.4 Bottom Sheets

Use `@gorhom/bottom-sheet` for the Add Food dialog and any long form flows. Half-screen height initially, expandable to 90%.

### 7.5 Barcode Scanner

Use `expo-barcode-scanner`. Request camera permission before showing scanner. On scan result → call `GET /api/food/barcode/:code`.

### 7.6 Image Upload (Profile Photo)

Use `expo-image-picker` to select image. Get a signed upload URL from `GET /api/uploads/request-url` (backend returns a pre-signed URL and object path). Upload directly to object storage via the signed URL. Then call `PUT /api/me/profile-photo` with the returned `objectPath`.

### 7.7 Push Notifications

Use `expo-notifications`. Request permission on Profile screen. Register device token with backend via `POST /api/me/push-subscriptions`. Supplement reminders are scheduled server-side and pushed via Web Push — in the Expo app, handle via FCM/APNs.

## 10. IMPORTANT BEHAVIOURAL DETAILS

1. **Date-driven everything:** All dashboard data is keyed to the selected date. When the user navigates left/right, all queries update simultaneously. Default to today's date on mount.

2. **Morning Check-In gate:** On load, the app checks `GET /api/me/morning-status/:today`. If sleep is not logged AND it's before noon, prompt a lightweight morning check-in modal on first open (not blocking, dismissable).

3. **Training load modal:** After any session is saved/updated, re-fetch training load. If `warnings.length > 0`, show the `TrainingLoadModal` — it is non-dismissable except via the Continue button. Use `sessionStorage`-equivalent (`AsyncStorage`) keyed to `load_warning_${date}` — only show once per date per session.

4. **Fight Camp override:** If a weight cut plan is active, the calorie targets returned by the API already include the fight-camp-adjusted numbers. The client does NOT need to re-calculate — always use values from `GET /api/me/profile` or the specific endpoint.

5. **Insufficient ACWR history:** When `insufficientHistory === true` in the training load response, display `—` for ACWR with "building baseline" subtext, and show the history notice. The backend handles suppressing misleading warnings automatically.

6. **Supplement intake toggle:** Supplement rows on Dashboard are real-time. Each tap instantly toggles the intake record and optimistically updates the UI, reconciling with the server response.

7. **Meal totals:** Display compact per-meal macro totals (kcal, P, C, F) at the bottom of each meal section. These are computed client-side from the food entries in that section.

8. **Onboarding suppression:** Once onboarding is complete (user has a profile), the onboarding screen must not be accessible except via "Recalculate Goals" from Profile. Check `user.onboardedAt` or equivalent.
