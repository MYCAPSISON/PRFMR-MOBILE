--------------------------------------------------------------------------------
  SECTION 1 — WHAT IS THIS APP?
--------------------------------------------------------------------------------

PRFMR is a combat sports performance tracking app. The web version already
exists at app.prfmr.link. This mobile app is a native iOS/Android version of
the same product, built so athletes can log their data, check their stats, and
track their weight cut from their phone — using the exact same account and data
as the web app.

The key requirement: users log in with the same credentials, and everything
they see is their REAL data from the server. There is no fake/test data in this
app. Every number shown comes from the backend at https://app.prfmr.link/api

--------------------------------------------------------------------------------
  SECTION 2 — THE TECHNOLOGY STACK (what tools are used)
--------------------------------------------------------------------------------

EXPO / REACT NATIVE
  The app is built with Expo, which is a framework on top of React Native.
  React Native lets you write JavaScript/TypeScript that runs as a real native
  iOS or Android app (not a website in a browser). Expo adds helpful tools on
  top: easy setup, camera access, secure storage, fonts, icons, etc.

  During development, you run the app in "Expo Go" — a free app on your phone
  that loads your code from the development server. You don't need to compile a
  full app to test changes.

EXPO ROUTER
  This handles navigation between screens. It works like a file system — the
  filename of each screen determines its URL/route. Files inside
  app/(tabs)/ become the bottom tabs. Files inside app/(auth)/ are the
  login/signup screens.

TYPESCRIPT
  All code is written in TypeScript, which is JavaScript with type checking.
  This means the editor catches mistakes before you run the app (e.g., passing
  a number where a string is expected). You'll see types defined everywhere
  (like "interface User { id: number; username: string; }"). These are just
  descriptions of what shape data has — they don't affect how the app runs,
  only help catch bugs while writing code.

REACT QUERY (@tanstack/react-query)
  This library handles all data fetching. Instead of manually writing
  fetch() calls everywhere, you use "useQuery" to fetch data and "useMutation"
  to send changes (like adding food). React Query automatically:
  - Shows loading states while data is fetching
  - Caches results so you don't re-fetch the same thing constantly
  - Retries failed requests
  - Refreshes stale data when the user comes back to a screen

  You'll see this pattern throughout every screen:
    const { data: targets } = useQuery({
      queryKey: ["targets", date],
      queryFn: () => apiFetch(`/me/targets/effective?date=${date}`),
    });

  "queryKey" is a unique name/ID for this piece of data. If you ever need to
  refresh data after saving something, you call:
    queryClient.invalidateQueries({ queryKey: ["targets", date] })
  This tells React Query "that data might be stale now, re-fetch it."

EXPO SECURE STORE
  This is how the app saves the login session between app launches. It's like
  a secure password vault built into the phone. All session cookies from login
  are stored here (as a JSON object called "prfmr_jar") so the user stays
  logged in even after closing the app.

DATE-FNS
  A utility library for working with dates. Used throughout the app for
  things like: formatting dates as "2026-04-16", adding/subtracting days,
  getting the start of a week, etc.

FEATHER ICONS & MATERIAL COMMUNITY ICONS
  Icon libraries. Used for all the small icons (arrows, checkmarks, moon icon
  for sleep, etc.).

--------------------------------------------------------------------------------
  SECTION 4 — AUTHENTICATION — THE FULL STORY (read this carefully)
--------------------------------------------------------------------------------

This was the hardest part of building the app. Here is the complete story,
including the bugs we had and how each was fixed.

─── HOW WEBSITES HANDLE LOGIN ───────────────────────────────────────────────

When you log into a website, the server creates a "session" and gives the
browser a tiny ID called a "session cookie." The browser stores this and sends
it back with EVERY future request automatically. This is invisible — the
browser handles it silently.

─── PROBLEM 1: Fetch API hides Set-Cookie on iOS ────────────────────────────

React Native's fetch() mimics the browser's security model, which deliberately
HIDES the Set-Cookie response header from JavaScript code. This is a browser
security feature (to prevent scripts from stealing HttpOnly cookies). React
Native copied it even though it doesn't need to on a native app.

Result: after login, response.headers.get("set-cookie") returned null. The
session cookie was invisible to our code. Subsequent API calls had no cookie
and got "401 Not Authenticated."

Fix: Use XMLHttpRequest (the older browser network API) for the login call.
React Native's XHR implementation does NOT have this restriction.
xhr.getAllResponseHeaders() returns all headers including Set-Cookie.

─── PROBLEM 2: The GAESA bug — single-cookie storage got overwritten ────────

Even after switching to XHR for login, the 401 errors continued. Diagnostic
logging revealed the actual sequence of events:

  1. App starts → loads "connect.sid=xxx" from storage → sends it to /user/me
  2. /user/me SUCCEEDS (200) → but its response contains Set-Cookie: GAESA=yyy
     (GAESA is a CSRF protection token the server sets on authenticated routes)
  3. Our code was saving GAESA as the session, replacing connect.sid entirely
  4. All subsequent requests sent Cookie: GAESA=yyy (without connect.sid)
  5. Server rejected with 401 — connect.sid was missing

The server sends MULTIPLE cookies. We were storing only the LAST one, which
overwrote the actual session cookie.

Fix: Implement a proper COOKIE JAR. The jar holds ALL cookies by name:
  { "connect.sid": "connect.sid=s%3Axxx", "GAESA": "GAESA=yyy" }

When the server sends Set-Cookie: GAESA=yyy, we ADD it to the jar without
removing connect.sid. When making requests, we send ALL cookies:
  Cookie: connect.sid=xxx; GAESA=yyy

─── THE CURRENT AUTH FLOW (how it works now) ────────────────────────────────

LOGIN:
  1. User enters email + password → loginWithXhr() is called
  2. XHR POST to /auth/login → server responds with session cookie(s)
  3. getAllResponseHeaders() exposes all Set-Cookie lines
  4. Each cookie is parsed and added to cookieJar (in memory)
  5. cookieJar is saved as JSON to SecureStore under key "prfmr_jar"
  6. Login response body has { user: {...} } → user is set immediately

EVERY REQUEST:
  7. apiFetch() reads cookieJar → builds "Cookie: connect.sid=xxx; GAESA=yyy"
  8. Sends that header with every request
  9. If the response has a new Set-Cookie, it ADDS it to the jar (never replaces)

APP RESTART:
  10. loadSession() reads "prfmr_jar" from SecureStore → cookieJar is restored
  11. /user/me is called with the saved cookies → if valid, user stays logged in

LOGOUT:
  12. POST /auth/logout → clearSession() → cookieJar cleared → SecureStore cleared

─── STORAGE KEY ─────────────────────────────────────────────────────────────

The cookie jar is stored under the key "prfmr_jar" in SecureStore.
(Old versions used "prfmr_session" — that key is no longer used.)

⭐⭐⭐ IMPORTANT — IF LOGIN BREAKS IN THE FUTURE:
  The diagnostic logs to look for after login:
    [auth] Session captured via XHR ✓ — cookies: connect.sid, GAESA
    [api] → /me/targets | cookies: [connect.sid, GAESA]
    [api] ← /me/targets 200

  If you see "cookies: []" (empty) for requests after login, the XHR cookie
  capture stopped working. Check:
  a) Whether getAllResponseHeaders() still returns Set-Cookie lines
  b) Whether the cookie names changed (the server might rename GAESA)
  c) Whether the server switched to a different auth mechanism (Bearer tokens)

--------------------------------------------------------------------------------
  SECTION 5 — HOW DATA FETCHING WORKS
--------------------------------------------------------------------------------

Every piece of data shown in the app goes through the same pipeline:

  1. COMPONENT calls useQuery() with a queryKey and a queryFn
  2. REACT QUERY checks its cache — if fresh data exists, shows it immediately
  3. If not cached (or stale), React Query calls the queryFn
  4. queryFn calls apiFetch() with the API path
  5. apiFetch() reads cookieJar → adds "Cookie: connect.sid=xxx; GAESA=yyy"
  6. apiFetch() calls fetch() to make the actual HTTP request to the server
  7. The server checks the cookies → confirms the user is logged in
  8. The server returns JSON data
  9. apiFetch() parses the JSON and returns it
  10. React Query stores it in cache and passes it to the component
  11. The component renders the data

When the user WRITES data (saves food, logs weight, etc.):
  1. Component calls useMutation()
  2. On success, it calls queryClient.invalidateQueries() on related queryKeys
  3. This marks the cached data as stale
  4. React Query re-fetches the fresh data from the server
  5. The screen updates with the new values

ALL API PATHS (the endpoints the app talks to):
  POST  /auth/login                     ← Login
  POST  /auth/logout                    ← Logout
  GET   /user/me                        ← Get current user info
  GET   /me/food/:date                  ← Food log for a date
  POST  /me/food                        ← Add a food entry
  DELETE /me/food/:id                   ← Remove a food entry
  GET   /me/targets/effective?date=     ← Calorie/macro targets for a date
  GET   /me/morning-status/:date        ← Has sleep/weight been logged today?
  PUT   /me/sleep/:date                 ← Log sleep hours + quality
  POST  /weights                        ← Log body weight
  POST  /me/rest-day/:date              ← Mark today as rest day
  DELETE /me/rest-day/:date             ← Unmark rest day
  GET   /me/readiness/:date             ← Training readiness score
  POST  /me/provisional-checkin         ← Quick how-do-you-feel check-in
  GET   /me/amqs/score/:date            ← AMQS readiness score
  GET   /me/weight-cut                  ← Active weight cut plan
  GET   /me/weights/range?start=&end=   ← Weight history for graph
  GET   /workouts/sessions?start=&end=  ← Training session history
  GET   /me/stacks/scheduled?date=      ← Scheduled supplement reminders
  GET   /me/supplement-intakes/:date    ← Which supplements taken today
  POST  /me/supplement-intakes          ← Mark supplement as taken
  DELETE /me/supplement-intakes/:id     ← Unmark supplement intake

--------------------------------------------------------------------------------
  SECTION 6 — THE SCREENS (what each tab does)
--------------------------------------------------------------------------------

DASHBOARD (app/(tabs)/index.tsx)
  The most complex screen. Shows everything for "today" with a date navigator.
  Contains these sections (each is its own component):
  - FightCampHero: Shows days until fight, current vs target weight, weekly
    progress. Only shows if user has an active weight cut plan.
  - MorningCheckIn: Prompts to log sleep, morning weight, and training.
    Disappears once all 3 are done.
  - ProvisionalCheckIn: "How do you feel today?" quick 3-question form.
  - AmqsCard: Shows the athlete's readiness score (0-100) for today.
  - MacroCards: Shows calories, protein, carbs, fat (consumed vs target).
  - WeightTrend: A simple line graph of weight over the past 4 weeks.
  - SupplementsPreview: Shows today's supplement reminders with check-off.
  - MealsList: Lists all food logged for the day with totals.

TRAINING (app/(tabs)/training.tsx)
  Shows recent workout sessions. Tapping one expands to show exercises/sets.

SUPPLEMENTS (app/(tabs)/supplements.tsx)
  Shows all supplement stacks with their scheduled reminders. Lets user check
  off supplements as taken.

NUTRITION (app/(tabs)/nutrition.tsx)
  Detailed food log for a selected date. Broken down by meal type (breakfast,
  lunch, dinner, snack). Allows adding and removing food entries.

WEIGHT CUT (app/(tabs)/weightcut.tsx)
  Detailed view of the active weight cut plan with weekly targets.

PROFILE (app/(tabs)/profile.tsx)
  Shows user info (username, email, sport, weight class). Logout button.

FEEDBACK (app/(tabs)/feedback.tsx)
  Simple form to submit bug reports or feature requests.

The app uses a DARK THEME matching the web app exactly. All colors are defined
in constants/colors.ts and accessed through the useColors() hook.

    const colors = useColors();
  Then use colors.primary, colors.card, etc. in your StyleSheet.
  Do NOT hardcode hex values in screen files — always use useColors().

  SECTION 8 — ⭐ THINGS TO PAY MOST ATTENTION TO ⭐
--------------------------------------------------------------------------------

These are the areas most likely to cause problems when you add new features
or debug issues. Read these carefully.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ⚠️  1. THE COOKIE JAR — lib/api.ts
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  The cookieJar object in lib/api.ts stores ALL session cookies. It must
  contain BOTH the session cookie ("connect.sid") AND any other cookies the
  server sends (like "GAESA" for CSRF protection).

  The most important rule: when a new Set-Cookie arrives, ADD it to the jar.
  NEVER replace the whole jar with just the new cookie.

  After login, the Expo console should show:
    [auth] Session captured via XHR ✓ — cookies: connect.sid, GAESA
    [api] → /me/targets | cookies: [connect.sid, GAESA]
    [api] ← /me/targets 200

  If any request shows "cookies: []" — the jar is empty and the session
  was lost. The user will need to log out and log back in.

  The jar is persisted in SecureStore under the key "prfmr_jar" as JSON.
  On the next app launch, loadSession() restores it automatically.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ⚠️  2. REACT QUERY KEYS — must be consistent
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  React Query uses the queryKey to identify and cache data. If you fetch
  something with queryKey: ["food", date] but then invalidate with
  queryKey: ["food"] (without the date), the cache won't update correctly
  and the user will see stale data.

  Rule of thumb: the queryKey must match EXACTLY between useQuery and
  invalidateQueries calls. Include all variables that affect the data
  (especially the date).

  Good example:
    useQuery({ queryKey: ["food", date], ... })
    queryClient.invalidateQueries({ queryKey: ["food", date] })  ✓

  Bad example:
    useQuery({ queryKey: ["food", date], ... })
    queryClient.invalidateQueries({ queryKey: ["food"] })  ✗ won't update

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ⚠️  3. DATE HANDLING — always use T12:00:00 timezone trick
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  When you write: new Date("2026-04-16")
  JavaScript interprets this as UTC midnight. If you're in a timezone BEHIND
  UTC, this can become the previous day locally — the wrong date!

  Fix: always add T12:00:00 when turning a "yyyy-MM-dd" string into a Date.

  Good:
    format(new Date(date + "T12:00:00"), "yyyy-MM-dd")
    addDays(new Date(date + "T12:00:00"), 1)

  Bad:
    format(new Date(date), "yyyy-MM-dd")  ← might shift by a day

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ⚠️  4. ENVIRONMENT VARIABLES — baked in at build time
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  The API URL (https://app.prfmr.link/api) is stored in EXPO_PUBLIC_API_URL.
  The "EXPO_PUBLIC_" prefix is important: Expo bakes this INTO the JavaScript
  bundle at compile time. If you change it, restart the Expo server (Metro)
  — a hot reload is NOT enough.

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  ⚠️  5. RELOADING vs. HOT RELOADING in Expo Go
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  HOT RELOAD (automatic): Small UI changes appear instantly. The in-memory
  cookieJar is preserved. Login stays active.

  FULL RELOAD (shake phone → Reload):
  This resets the JavaScript context. The in-memory cookieJar is cleared.
  BUT: loadSession() runs on startup and restores cookieJar from SecureStore.
  So you stay logged in even after a full reload.

  When to do a full reload:
  - After changing environment variables (EXPO_PUBLIC_*)
  - After changing native configuration (app.json)
  - When the app seems stuck in a weird state
  - After updating packages

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  To add a new tab screen:
  1. Create: app/(tabs)/myscreen.tsx
  2. Add to tab bar: app/(tabs)/_layout.tsx (follow existing tab patterns)
  3. Add an icon from Feather or MaterialCommunityIcons
  4. Use useColors() for all colors

  ⚠️  7. ADDING A NEW API CALL
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  To fetch data:
    const { data, isLoading } = useQuery({
      queryKey: ["my-key", variable],
      queryFn: () => apiFetch(`/me/endpoint?param=${variable}`),
    });

  To send data:
    const qc = useQueryClient();
    const mut = useMutation({
      mutationFn: (val: number) =>
        apiFetch("/me/endpoint", { method: "POST", body: { val } }),
      onSuccess: () => qc.invalidateQueries({ queryKey: ["my-key", variable] }),
    });
    // Trigger with: mut.mutate(42)

  You NEVER need to change lib/api.ts to add a new endpoint — apiFetch()
  is generic and works with any path automatically.

--------------------------------------------------------------------------------
  SECTION 9 — DEBUGGING TIPS
--------------------------------------------------------------------------------

THE EXPO DEV CONSOLE (shake phone → look at log output):
  The app logs every API request in development mode:
    [api] → /path | cookies: [connect.sid, GAESA]
    [api] ← /path 200 | new cookie: NO

  If you see 401 errors, check whether cookies are listed before the request.
  If cookies: [] — the jar is empty. The user needs to log in again.

COMMON ERROR MESSAGES:
  "Not authenticated" (401)  ← Cookie jar is empty or cookies are wrong.
                                Log out and back in.
  "Cannot reach the server"  ← No internet, or server is down.
                                Test: https://app.prfmr.link/api/user/me
  "Unexpected server response" ← The endpoint URL is wrong or the server
                                 returned HTML (error page) instead of JSON.

CHECKING IF THE API IS ONLINE:
  Open any browser: https://app.prfmr.link/api/user/me
  You should see: {"message":"Not authenticated"} (correct — server is up)
  If you see a connection error, the server is down.

CHECKING THE COOKIE JAR STATE:
  After login, look for this in the Expo logs:
    [auth] Session captured via XHR ✓ — cookies: connect.sid, GAESA
  Then for subsequent requests:
    [api] → /me/targets | cookies: [connect.sid, GAESA]
  If the cookies list shows both names — auth is working correctly.

--------------------------------------------------------------------------------
  SECTION 11 — HOW TO MAKE CHANGES SAFELY
--------------------------------------------------------------------------------

BEFORE CHANGING lib/api.ts:
  This file is the most critical. Any mistake breaks ALL data loading.
  Always test: login → data loads → logout → login again after any change.
  The key function is apiFetch() and the cookie jar logic. Be very careful
  not to accidentally clear cookieJar or reset it incorrectly.

BEFORE CHANGING context/AuthContext.tsx:
  This controls whether the user is logged in. A mistake could cause:
  - Infinite redirect loops (login → dashboard → login → ...)
  - Users being logged out unexpectedly
  - The login button not working
  Always test the full flow: login, use the app, logout, login again.

BEFORE CHANGING app/_layout.tsx:
  This is the root wrapper for the entire app. Breaking it breaks everything.
  Changes here are rare — only for adding new global providers.

FOR MOST FEATURES (adding UI, changing screen content):
  Files in app/(tabs)/ are relatively safe to change. They only affect
  one screen. You can change freely and test just that tab.

KEEP THE OVERVIEW FILE UPDATED:
  Whenever you make a significant change to how the app works (especially auth,
  data fetching, or adding new screens), update this file to reflect it.
